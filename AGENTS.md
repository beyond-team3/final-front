# AGENTS.md

이 파일은 이 저장소에서 프론트엔드 수정 작업을 수행할 때 따라야 하는 기본 작업 지침이다.

## 기본 원칙
- 요청받은 범위 안에서만 수정한다.
- 현재 작업과 무관한 기능, 화면, 공용 로직은 웬만하면 건드리지 않는다.
- 불가피하게 영향 범위가 넓어지는 경우, 왜 필요한지 먼저 정리하고 최소 범위로 제한한다.
- 기존 디자인 시스템, 컴포넌트 구조, 상태 관리 방식이 있다면 우선 존중한다.

## 작업 시작 전
- 관련 화면, 컴포넌트, 라우트, 상태 저장소, API 호출 위치를 먼저 확인한다.
- 수정 대상과 연관된 파일만 선별해서 읽고, 추측만으로 넓게 변경하지 않는다.
- 동일 기능과 연결된 기존 구현 패턴이 있으면 먼저 재사용 가능 여부를 확인한다.

## 수정 작업 규칙
- 한 번에 큰 구조를 갈아엎기보다, 요구사항을 만족하는 최소 변경을 우선 적용한다.
- UI 수정 시 스타일만 바꾸면 되는지, 마크업 변경이 필요한지, 상태 로직 수정이 필요한지 구분해서 작업한다.
- 기능 수정 시 부수효과가 생길 수 있는 공통 컴포넌트, 유틸, 전역 상태 변경은 특히 신중하게 다룬다.
- 디버깅용 코드(`console.log`, 임시 주석, 죽은 코드)는 남기지 않는다.
- 사용하지 않는 import, 변수, 함수는 정리한다.

## 테스트 및 검증
- 작업 후 가능한 범위에서 직접 실행 또는 빌드로 검증한다.
- 최소한 변경한 기능의 진입 경로와 주요 상호작용을 직접 확인한다.
- 영향받을 수 있는 인접 기능도 함께 점검해 회귀 여부를 확인한다.
- 테스트나 실행을 못 했으면, 이유와 남아 있는 검증 리스크를 명확히 남긴다.

## 변경 보고
- 무엇을 왜 수정했는지 짧고 명확하게 정리한다.
- 사용자가 알아야 할 영향 범위, 제약사항, 후속 확인 포인트가 있으면 함께 남긴다.

## 커밋 규칙
- 작업이 완료되면 반드시 루트의 [`COMMIT_PROMPT.md`](/home/hakyung02/BeyondSW/final/final-front/COMMIT_PROMPT.md)를 참조해서 커밋한다.
- 커밋은 변경사항을 의미 있는 단위로 나눈다.
- 서로 다른 목적의 변경을 한 커밋에 섞지 않는다.
- 커밋 메시지 형식과 분리 기준은 `COMMIT_PROMPT.md`의 규칙을 그대로 따른다.
- 커밋 시 `LINEAR_ID`는 기본적으로 `FRT-{현재 브랜치명 끝의 숫자}` 형식으로 자동 지정한다. 예: 브랜치가 `test/api-v2-frt-87`이면 `FRT-87`.
- 현재 브랜치명 끝에 숫자가 없을 때만 `COMMIT_PROMPT.md` 규칙에 따라 플레이스홀더 또는 사용자 입력 기준으로 처리한다.

## 금지 사항
- 요청하지 않은 리팩터링으로 작업 범위를 키우지 않는다.
- 관련 없거나 검증하지 못한 수정사항을 끼워 넣지 않는다.
- 동작 확인 없이 완료 처리하지 않는다.
